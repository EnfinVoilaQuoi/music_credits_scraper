# Plan — REFACTORING.md phases 1-4 + AUDIT §8.1-8.3 (providers, SQLAlchemy+observations, asyncio)

## Contexte

L'AUDIT (2026-07-10) a identifié la dette du cœur historique : `data_enricher.py` (1 796 l., `enrich_track()` connaît intimement 7 sources et leur ordre), `data_manager.py` (1 804 l., 87 accès positionnels `row[N]`, 44 placeholders, ALTER TABLE en try/except), `track.py` (945 l., défensif). La cible §8 : architecture provider avec `Observation` + moteur de réconciliation central (vote BPM généralisé), SQLAlchemy 2.0 + Alembic avec table `observations(track_id, field, value, source, confidence, seen_at)`, et un modèle de concurrence unique asyncio. Le préalable (phase 0 : Black/Ruff 0 signalement, 238 tests, CI) est **fait**.

**Règle d'or** (REFACTORING.md) : refacto à comportement constant, chaque étape commitable, l'app tourne entre chaque commit. Migration incrémentale, pas de réécriture.

**Décisions ratifiées par l'utilisateur (2026-07-11)** :
- **SQLAlchemy Core + dataclasses** : `MetaData`/`Table` dans `src/persistence/schema.py`, requêtes nommées ; `Track`/`Artist` restent LE modèle domaine (la GUI ne change pas) ; le mapper `track_from_row` est la couture ORM↔domaine. Pas d'ORM déclaratif (sessions vs 18 workers threads = champ de mines).
- **Providers sync d'abord** : 3 passes par provider — extraction sync (C), `fetch() -> list[Observation]` (E5), `async def` (F). Jamais structure + concurrence dans le même commit.
- REFACTORING §1.3 (migrations `user_version`) **supprimé** — absorbé par Alembic (note AUDIT §7). Corollaire : **gel du schéma SQL** jusqu'à la phase E.
- Concept `Observation` introduit en phase E (avec sa persistance), pas avant ; `BpmBallot` (phase C) en est l'embryon.

**Amendements Phase E ratifiés (2026-07-13, revue pré-implémentation)** :
- La phase A a finalement livré des migrations `user_version` (46 entrées, `src/utils/db.py`) : le bootstrap Alembic doit les absorber et rester **permanent** (un backup pré-Alembic restauré via `database_backup.py` doit garder un chemin de migration après E3).
- `observations` = modèle **upsert** avec `UNIQUE(track_id, field, source)` (dernière valeur par source, `seen_at` = dernière vue), pas d'historique append-only — rien ne le consomme.
- **key/mode = deux observations séparées** (`field='key'` / `field='mode'`), mais traitées en **paire** par le moteur : l'un sans l'autre = donnée incomplète, non fiable seule.
- L'**arrêt de l'écriture legacy déménage de E6 vers E7** (E6 = bascule lecture pure, réversible) ; **`NullPool`** à E2 (comportement constant : une connexion par opération, comme aujourd'hui).
- `PRAGMA foreign_keys` n'a jamais été activé dans l'app → pas de cascade : `delete_track`/`merge_tracks` gèrent les observations **explicitement**.
